# Zo Cookbook — New Idea Quality Standard

## Definition
One **idea** is one complete cookbook card: the named concept plus its category, description, build notes, difficulty, monetization/use case, and action/handoff copy generated by the app. A single field inside a card does not count as an idea.

## Schema rule
For new **app-style ideas**, use the same fields already used by `App` entries in `src/data/cookbook-data.ts`:

- `Category`
- `What it does`
- `How to build it on Zo`
- `Monetization`
- `Difficulty`

Do not invent a separate `Handoff brief` field for app-style batches. The app already generates a Zo handoff from the standard fields. Only add new fields if we intentionally change the schema and UI.

## Category rule
Use existing cookbook categories unless there is a strong product reason to add a new one. New categories must stay inside what Zo can actually do: files, browser/web research, integrations, automations, agents, zo.space, Zo Sites, services, datasets, Stripe, messaging, and workspace-local tools.

Bad fit: “Zo as an Uber driver.”
Good fit: “Vacation Price Scout” that compares hotels/motels, saves options, tracks prices, and sends a trip-planning brief.

## Voice match
New cards should sound like the existing site:

- Concrete, direct, and buildable.
- Uses the existing labels: “What it does,” “How to build it on Zo,” “Monetization,” and “Difficulty.”
- Shows the Zo-native path: Space, Sites, Services, agents, datasets, integrations, files, browser tools, and scheduled workflows.

## Quality bar
Each new idea must meet or beat the existing card quality.

Minimum standard:

1. **Mind-expanding** — not just “summarize my email” or “make a dashboard.” It should reveal a new use case for a personal cloud computer with an agent inside it.
2. **Zo-native** — depends on something Zo uniquely combines: files + tools + automations + hosting + integrations + user-owned runtime.
3. **Actionable** — a user can hand it to Zo and get a first version built.
4. **Specific** — names inputs, outputs, storage, routes, schedules, and tools where possible.
5. **Handoff-ready** — includes a recipe/spec/prompt that can be copied into Zo.

## Required fields for new batch markdown

Each app-style entry should use this structure:

```md
### 001. Idea Name
**Category:** Existing app category
**What it does:** ...
**How to build it on Zo:** ...
**Monetization:** ...
**Difficulty:** Beginner | Intermediate | Advanced
```

## Critique filter
Cut or rewrite anything that fails one of these tests:

- Could this just be a ChatGPT prompt? If yes, it is not Zo Cookbook material.
- Would a new Zo user understand why owning the computer matters here?
- Does “How to build it on Zo” include enough concrete implementation detail for the app’s generated handoff to work?
- Is the idea actually useful, weirdly delightful, or strategically revealing?
- Is the copy in the same plainspoken style as the current site?
